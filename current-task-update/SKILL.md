# 当前任务更新

维护 `current-task/README.md`，把它作为当前项目任务的简洁交接记录。

## 输入

把用户调用技能时附带的文字当作更新参数。参数可以描述：

- 任务重点或标题
- 本次发生了什么变化
- 需要强调的内容
- 测试环境和命令
- 已知风险、阻塞点或下一步
- 是追加、刷新，还是总结当前会话

如果用户没有提供额外参数，就根据当前会话、当前 Git 变更和最近验证结果推断要更新的内容。

## 文件规则

- 始终使用相对路径 `current-task/README.md`。
- 如果 `current-task/README.md` 不存在，就创建它。
- 如果文件已经存在，就原地更新。
- 如果 `current-task/` 目录不存在，就创建它。
- 除非用户明确要求，不要写入绝对路径的任务记录文件；本项目标准是相对路径。
- 不要删除仍然有价值的历史上下文，除非它已经过期或被新的已验证事实推翻。

## 必须流程

1. 如果当前回合还没有满足工作区启动阅读要求，先按工作区规则完成启动阅读。
2. 如果 `current-task/README.md` 已存在，先读取现有内容。
3. 只检查本次更新所需的最小上下文：
   - 当前会话摘要（如果可用）。
   - 相关仓库的 `git status --short`。
   - 必要时查看 `git diff --stat` 或指定文件 diff。
   - 如果命令或测试结果属于任务状态的一部分，也要纳入。
4. 用简洁、最新、可交接的内容更新 `current-task/README.md`。
5. 写入后复查文件内容。
6. 如果复查后的任务状态不是“已完成”，不要把技能执行停在任务记录更新上；应立即继续执行 `current-task/README.md` 中记录的下一步或当前任务，直到任务完成、遇到真实阻塞，或用户明确要求暂停。

## 内容规范

默认使用中文，除非现有文件明显是英文。

适用时优先使用这些章节：

```markdown
# 当前任务记录

## 任务

## 当前变更

## 目标行为

## 验证重点

## 测试环境

## 当前状态

## 风险与注意点

## 下一步
```

不要强行保留所有章节，只保留有价值的章节。

## 更新策略

- 新任务：创建清晰的初始记录，包含任务目标、范围和预期验证方式。
- 已有任务：把新事实合并到正确章节，不要盲目追加重复内容。
- 长会话：把会话总结成稳定事实，包括决策、执行过的命令、测试结果、失败、修复和当前状态。
- 基于 Git 的更新：记录仓库和路径，但避免写成冗长的逐文件流水账。
- 验证信息：必要时记录精确的目标环境、内核/版本标识、命令和通过/失败结果。
- 不确定信息：明确标记为待确认或未验证。

## 质量要求

- 另一个代理应该能在不阅读完整会话的情况下接手任务。
- 内容要简洁，但必须足够表达任务意图、当前状态、验证证据和下一步。
- 重要的日期、版本、主机、路径和命令名必须保留。
- 如果任务已经完成，要在 `当前状态` 中明确说明。
- 如果任务没有完成，要在 `当前状态` 或 `下一步` 中明确可立即执行的动作，并在更新后直接开始执行该动作。
